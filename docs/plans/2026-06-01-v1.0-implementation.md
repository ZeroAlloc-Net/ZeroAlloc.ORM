# ZeroAlloc.ORM v1.0 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `1.0.0` — the API-freeze release. No new features; only polish, docs site, selected backlog cleanups, and final hygiene before the public API lock holds for v1.x.

**Architecture:** v1.0 is explicitly a **polish milestone** per design Section 5. The public API surface was already frozen in v0.7 Phase D (`PublicAPI.Shipped.txt`, 103 entries / 16 types). The release gates are:

- **G1 (cookbook)** — every design Section 5 recipe has a dedicated, high-quality page in `docs/cookbook/`. Six recipes already shipped across v0.3-v0.6; backfill the two gaps (dedicated `flat-row.md`, `provider-quirks.md`) and audit the existing six.
- **G2 (Docusaurus website)** — `https://orm.zeroalloc.net` via the shared **ZeroAlloc.Website** monorepo at `c:/Projects/Prive/ZeroAlloc/ZeroAlloc.Website/`. Mirrors the canonical per-package pattern (`apps/docs-<name>/` + `repos/<name>/`); no local Docusaurus scaffolding in this repo.
- **G3 (selected polish cleanups)** — four high-value, low-risk CLN items: v0.7-CLN2 (bench async-parity), v0.5-CLN1 (ZAO050 tightening), v0.4-CLN6 (ZAO062 per-element span), v0.4-CLN5 (Batch property diagnostic). Larger CLNs (recursive composites, nullable RT composite binding, pipeline arch refactor) defer past v1.0 — they don't block the freeze.
- **G4 (release hygiene)** — final README polish, badge audit, release notes consolidating the v0.1→v1.0 journey, conventional-commit verification across all merges.
- **G5 (tag 1.0.0)** — release-please proposes 1.0.0; pack-push publishes to NuGet.

**Tech Stack:**
- Docusaurus via the shared **ZeroAlloc.Website** monorepo (pnpm + turbo); per-package app at `apps/docs-orm/` pulling from `repos/orm/docs/`. Mirror `apps/docs-mediator/`, `apps/docs-rest/`, etc.
- Existing BDN harness for the async-parity capture.
- Existing PublicApiAnalyzers (locks the surface — any v1.0 PR that changes Shipped.txt is a breaking change).

**Authoritative reference:** [`docs/design/2026-05-30-v1.0-design.md`](../design/2026-05-30-v1.0-design.md) Section 5 (v1.0 milestone: "API freeze, polish, doc-link finalization"), Section 5 line 644 (cookbook list).

**Working backlog:** [`docs/plans/za-orm-backlog.md`](za-orm-backlog.md) — v1.0-G1/G2 + carry-forward CLN status.

**Pace:** ~7 days single-developer pace. ~17 tasks across 5 phases. Polish milestone — no new emit shapes, no public API additions.

**Lessons baked in (carried from v0.1-v0.7):**

1. Conventional-commit prefixes enforced by the v0.6-shipped PR-title lint workflow.
2. `PublicAPI.Shipped.txt` is frozen — any v1.0 PR that touches it is a breaking change that needs explicit reviewer sign-off.
3. Cap each agent dispatch at ~5-6 commits.
4. Sibling-repo cross-check first (look at AdoNet.Async's Docusaurus setup before scratch-writing).

## What v1.0 deliberately does NOT do

- **New emit shapes** — public API is frozen.
- **Breaking API changes** — would push to v2.0.
- **Recursive composites** — v0.5-CLN3, stays past v1.0.
- **Pipeline architecture refactor** — v0.4-CLN1, too invasive for the freeze release.
- **Nullable reference-type composite binding** — v0.5-CLN2.
- **SQL Server integration fixture** — defer until adopter demand.
- **ORM v2 features** — array params, TVPs, BulkCopy semantics, runtime provider routing — design doc explicitly punts these to v2.
