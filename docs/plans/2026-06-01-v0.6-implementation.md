# ZeroAlloc.ORM v0.6 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `0.6.0` — Postgres integration fixture (resolves accumulated v0.3/v0.4/v0.5 deferrals), full diagnostics catalog audit + per-code reference pages, ZA.Telemetry observability cookbook recipe, and selected backlog cleanup.

**Architecture:**

1. **Postgres integration fixture** via Testcontainers — completes the integration coverage map that Sqlite couldn't fill:
   - v0.3-CLN3: `IAsyncDbConnection.CanCreateBatch` forwarding (Sqlite reports false even though SqliteConnection ≥9.x supports it; Postgres via Npgsql properly forwards).
   - v0.4 stored-procedure round-trips (Sqlite has no native sprocs; Postgres `CREATE PROCEDURE` covers it).
   - v0.5 `[Materialize(Factory)]` Money.FromStorage end-to-end against a real `NUMERIC` column type.
   - Named-tuple output parameters on real sproc with `INOUT`/`OUT`.

2. **Diagnostics catalog audit** — every ZAO descriptor:
   - Has a `docs/diagnostics/ZAOxxx.md` reference page (most already exist; backfill missing).
   - `helpLinkUri` resolves to the published location.
   - Has at least one positive test (fires when expected) and one negative test (doesn't fire when not expected).
   - Catalog table in `README.md` lists every code.

3. **ZA.Telemetry composition cookbook** — adopters wire observability AT the consumer seam via the existing `[Instrument]` pattern from ZA.Telemetry. ZA.ORM ships NO `ActivitySource` (confirmed v0.6 design pivot per backlog). Cookbook shows the composition.

4. **Backlog cleanup** — opportunistically resolve high-value v0.3/v0.4/v0.5 cleanup items that align with v0.6 work.

**Tech Stack:**
- Testcontainers (Postgres) for the integration fixture.
- Existing snapshot + compile-smoke harness.
- Existing `pack-push.yml` for the v0.6.0 release.

**Authoritative reference:** [`docs/design/2026-05-30-v1.0-design.md`](../design/2026-05-30-v1.0-design.md) — Section 4 (diagnostics catalog), Section 4.5 (observability composition), Section 5 (v0.6 milestone scope: lines 596-598).

**Working backlog:** [`docs/plans/za-orm-backlog.md`](za-orm-backlog.md) — v0.6 entries (T2, T4, T5) + accumulated CLN items.

**Pace:** ~5-6 days single-developer pace. ~16 tasks across 5 phases. Design said 3-4 days; the +2 days come from Postgres fixture work that wasn't in the original v0.6 scope (it kept being kicked from v0.3/v0.4/v0.5 into v0.6).

**Lessons baked in (carried from v0.1-v0.5):**

1. Cookbook docs adopter-facing; no internal jargon.
2. Cross-link diagnostic docs.
3. Cap each agent dispatch at ~5-6 commits.
4. `helpLinkUri` MUST resolve to a real markdown file (no broken links shipped).
5. Reuse existing helpers (`BuildConnectionPrologue`/`Epilogue`, `EmitParameterBindingWithIndent`, etc.).

```csharp
public sealed class PostgresFixture : IAsyncLifetime
{
    private readonly PostgreSqlContainer _container;
    public IAsyncDbConnection Connection { get; private set; } = null!;
    public string ConnectionString => _container.GetConnectionString();

    public PostgresFixture()
    {
        _container = new PostgreSqlBuilder()
            .WithImage("postgres:16-alpine")
            .Build();
    }

    public async Task InitializeAsync()
    {
        await _container.StartAsync();
        var conn = new NpgsqlConnection(_container.GetConnectionString());
        Connection = conn.AsAsync();
        await Connection.OpenAsync(default).ConfigureAwait(false);
    }

    public async Task DisposeAsync()
    {
        await Connection.CloseAsync().ConfigureAwait(false);
        await _container.DisposeAsync();
    }

    public Task ExecuteDdlAsync(string sql) { ... }
}
```

**CI note:** GitHub-hosted runners support Docker; verify the test runs in `build-test` workflow. If it doesn't, gate behind `[Trait("Provider", "Postgres")]` and set an opt-in flag.
