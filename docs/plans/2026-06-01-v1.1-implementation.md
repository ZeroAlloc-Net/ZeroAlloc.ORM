# ZeroAlloc.ORM v1.1 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `1.1.0` ‚Äî minimal SQL migration runner inside `ZeroAlloc.ORM`. Unblocks ZA.Templates' full EF Core ‚Üí ZA.ORM swap.

**Architecture:**

A focused, additive feature. The public API surface frozen in v0.7 stays byte-identical for existing types; v1.1 adds new public types under a `ZeroAlloc.ORM.Migrations` namespace.

- **`MigrationRunner`** ‚Äî orchestrates apply. Takes an `IAsyncDbConnection`, an `IMigrationSource`, and an `IMigrationDialect`. Acquires a provider-specific advisory lock, reads applied versions from `__zaorm_migrations`, applies each newer script in a transaction, records the version. Idempotent: re-running with the same scripts is a no-op.

- **`IMigrationSource`** ‚Äî yields ordered `Migration` records (`int Version`, `string Name`, `string Sql`). Default impl `EmbeddedResourceMigrationSource` reads from an assembly's embedded resources matching a pattern like `Migrations/001_initial.sql`.

- **`IMigrationDialect`** ‚Äî provider-specific behaviour (CREATE TABLE for the history table, advisory lock acquisition, transaction semantics). Two implementations ship in v1.1: `SqliteMigrationDialect` and `PostgresMigrationDialect`.

- **`Migration` record** ‚Äî `int Version`, `string Name`, `string Sql`, optional `DateTime AppliedAt` after apply.

**Naming convention:** migration files are named `NNN_description.sql` where `NNN` is a 3+-digit version number. Versions must be strictly increasing. Gaps are allowed (e.g. 001, 003 with 002 missing) but result in 002 being permanently un-appliable.

**Locking strategy:**

- Postgres: `pg_advisory_lock(<constant>)` at runner entry, released at exit. Lets multi-instance API startup coordinate.
- Sqlite: `BEGIN EXCLUSIVE TRANSACTION` per migration. Sqlite's single-writer model means concurrent apply attempts serialize naturally.
- Other providers (SQL Server, MySQL): out of scope for v1.1 ‚Äî file as v1.1-CLN1.

**Transaction strategy:**

- Each migration applies in its own transaction. If migration N fails, N-1 is committed, N is rolled back, the runner throws ‚Äî adopter fixes and re-runs.
- An option to disable transactions (for migrations that contain DDL the provider doesn't allow in a transaction, e.g. some Postgres CONCURRENTLY operations) is OUT of scope for v1.1.

**Tech Stack:**
- New code lives in `src/ZeroAlloc.ORM/Migrations/` (under the existing `ZeroAlloc.ORM` package; no new NuGet package).
- New public types added to `PublicAPI.Unshipped.txt`; promoted to `Shipped.txt` at end of milestone.
- Integration tests use the existing Sqlite + Postgres fixtures from v0.6.

**Authoritative reference:** Migration runner design discussed inline in this plan; no separate design doc since the scope is small.

**Working backlog:** [`docs/plans/za-orm-backlog.md`](za-orm-backlog.md) ‚Äî v1.1 entry to add post-merge.

**Pace:** ~5 days single-developer pace. ~14 tasks across 5 phases.

**Lessons baked in (carried from v0.1-v1.0):**

1. TDD per task.
2. Conventional-commit prefixes enforced by the PR-title lint workflow.

### Task A.3 ‚Äî `MigrationRunner` + Sqlite end-to-end test

**Files:**
- New: `src/ZeroAlloc.ORM/Migrations/MigrationRunner.cs`:
  ```csharp
  public sealed class MigrationRunner
  {
      private readonly IAsyncDbConnection _connection;
      private readonly IMigrationSource _source;
      private readonly IMigrationDialect _dialect;

      public MigrationRunner(IAsyncDbConnection connection, IMigrationSource source, IMigrationDialect dialect);

      public async Task<IReadOnlyList<Migration>> RunAsync(CancellationToken ct = default);
  }
  ```

Algorithm:
1. Acquire dialect lock.
2. Ensure history table exists (CREATE TABLE IF NOT EXISTS).
3. SELECT applied versions.
4. Read source migrations, filter to those NOT in applied set.
5. For each pending migration in version order:
   - BEGIN TRANSACTION.
   - Execute migration SQL (may contain multiple statements).
   - INSERT version + name + UTC now into history table.
   - COMMIT.
6. Release dialect lock.
7. Return the list of migrations applied this run.

Tests:
- `tests/ZeroAlloc.ORM.Integration.Tests/Migrations/MigrationRunnerSqliteTests.cs`:
  - `Empty_db_applies_all_migrations`.
  - `Already_applied_migration_is_skipped` (idempotent re-run).
  - `Mid_apply_failure_rolls_back_failing_migration_only` (migration 002 fails; 001 stays committed; 002 not in history; 003 not attempted).
  - `Out_of_order_versions_are_applied_in_version_order_not_source_order`.
