# ZeroAlloc.ORM v0.7 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `0.7.0` — benchmark suite (BenchmarkDotNet) comparing hand-written ADO.NET / Dapper.AOT / ZeroAlloc.ORM, ZA.Rest collision smoke test (the v1.0 gate), README + Quick Start polish mirroring the AdoNet.Async / ZA.Rest pattern, and a v1.0-blocking API review pass.

**Architecture:**

1. **Benchmark suite** at `tests/ZeroAlloc.ORM.Benchmarks/` using BenchmarkDotNet:
   - **Baselines:** hand-written ADO.NET (zero-overhead reference) + Dapper.AOT (closest competitive AOT-clean library).
   - **Workloads:** single-row read, multi-row read, head+lines (multi-result-set), simple INSERT.
   - **Backends:** Sqlite in-memory (CI-friendly) AND Postgres via Testcontainers (real-provider).
   - **Outputs:** Markdown report committed under `docs/benchmarks/` for adopter visibility.

2. **ZA.Rest collision smoke** at `tests/ZeroAlloc.ORM.GeneratorCollision.AotSmoke/` (the v1.0 release gate):
   - One project references both `ZeroAlloc.Rest.Generator` AND `ZeroAlloc.ORM.Generator`.
   - Uses both attributes on adjacent partial classes (e.g. a REST controller calling an ORM repository).
   - AOT-publishes cleanly.
   - `.github/workflows/collision-smoke.yml` runs on every PR.

3. **README + Quick Start polish** mirroring AdoNet.Async / ZA.Rest:
   - Packages table with AOT-compatibility column.
   - Quick Start showing the 4 most common annotation shapes (`[Query]`, `[Command]`, `[StoredProcedure]`, `IAsyncEnumerable`).
   - "NativeAOT compatibility" section.

4. **API review pass** — walk every public type in `ZeroAlloc.ORM.Abstractions` + `ZeroAlloc.ORM` and confirm:
   - No accidental publics. Use `internal` aggressively for non-adopter-facing types.
   - Naming consistency.
   - Snapshot the public API surface (PublicAPI.txt or Verify.NET-style) for v1.0 lock.

**Tech Stack:**

- `Dapper.AOT` for the comparative baseline.

**Authoritative reference:** [`docs/design/2026-05-30-v1.0-design.md`](../design/2026-05-30-v1.0-design.md) — Section 5 (v0.7 milestone scope at line 599).

## Risk register

| Risk | Mitigation |
|---|---|
| Dapper.AOT API doesn't match our access patterns 1:1 | Pick the closest comparable; document the mapping in the benchmark file's comments |
| Postgres benchmark CI runtime explosion | Gate Postgres workloads behind a name filter; CI runs Sqlite-only |
| ZA.Rest's generator collides with ours at the partial-method or assembly-attribute level | Investigate early in Phase B.1; if a real collision surfaces, file it as the EXACT v1.0 blocker and address before tag |
| API review finds many accidental publics requiring breaking changes | If breaking changes surface, ship as `1.0.0-preview` or defer the rename to v1.0; don't break adopters in a minor release |
| BenchmarkDotNet's results vary widely run-to-run | Commit a representative set; emphasize that benchmarks measure relative position, not absolute throughput |
| API overload during long agent dispatches | Cap at ~5-6 commits per agent run |
