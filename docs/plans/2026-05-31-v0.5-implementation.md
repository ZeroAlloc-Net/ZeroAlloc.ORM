# ZeroAlloc.ORM v0.5 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `0.5.0` — multi-column composite materialization (`Money(decimal Amount, string Currency)` style), composite parameter binding, `[Materialize(Factory = "...")]` resolution, nullable composite handling, and a Sqlite-side `decimal`-as-text adopter recipe.

**Architecture:** Two changes lit up across the existing pipeline.

1. **Composite materialization** — `ConventionDiscovery` already recognises a `MultiArgCtor` `ConventionKind` (Section 3 of the design). Today it returns the resolution but the emit treats it as `Unknown`. v0.5 wires it through: detect a multi-arg-ctor type as a composite, expand its column count beyond the C# ctor arity, and emit one `reader.GetXxx(ord)` per inner column. The expanded column count cascades to outer FlatRow/DomainEntity positions so `record OrderRow(int Id, Money Total)` reads as 3 columns (`int`, `decimal`, `string`).

2. **`[Materialize(Factory)]` resolution** — the attribute already exists in Abstractions; v0.5 makes it active. Generator looks up the named `static` factory method on the annotated type, maps its parameter list to columns by name, emits `T.FactoryName(reader.GetXxx(ord), ...)`. Conflicts with discovery rules covered by ZAO031 / ZAO043 / ZAO044 (already in the catalog from v0.2).

**Composite binding** is the mirror image: a method parameter typed `Money total` unpacks into two SQL parameters `@total_Amount` + `@total_Currency`. Positional unpacking — no name-coupling to ctor args, so the SQL author picks the parameter-name suffix scheme.

**Nullable composite handling** is the trickiest piece. `Money? Total` means "if ALL composite columns are DBNull → null; if ANY are DBNull → throw + ZAO050 compile-time warning that this case is undetectable at compile time."

**Tech Stack:** Same as v0.4. No new substrate work. Heavy use of the existing `ConventionDiscovery.Resolve` API + `PrimitiveCatalog`.

**Authoritative reference:** [`docs/design/2026-05-30-v1.0-design.md`](../design/2026-05-30-v1.0-design.md) — Section 3 (convention discovery), Section 3.5 (nullable composite handling, line 330), Section 4 (diagnostic catalog: ZAO043, ZAO050).

**Working backlog:** [`docs/plans/za-orm-backlog.md`](za-orm-backlog.md) — v0.5 entries (v0.5-T1 through T4).

**Pace:** ~1 week single-developer pace. ~14 tasks across 6 phases. TDD per task with snapshot + compile-smoke + integration tests where applicable.

**Lessons baked in (carried from v0.1-v0.4):**

1. Snapshot + compile-smoke for every new emit shape variant.
2. +15-20% per-task budget for analyzer friction (MA0048, MA0004, MA0006, CA1707, EPS06).
3. `@`-prefix verbatim safety on every emitted identifier reference (including the new `@composite_subfield` parameter names).
4. No `Location`/`Compilation`/`SyntaxNode`/`ISymbol` in cached generator model types — `EquatableArray<T>` for collections.
5. Reuse the v0.4 helpers (`BuildConnectionPrologue`/`Epilogue`, `BuildScalarConvertExpression`, `EmitParameterBindingWithIndent`, `BuildCommandTextAssignment`) — DO NOT duplicate.
6. Bundle related tasks per dispatch but cap at ~5-6 commits per agent run.

## Phase B — Composite parameter binding (Days 3-4)

### Task B.1 — Detect composite binding on method parameter

**Files:**
- Modify: `OrmGenerator.cs` — parameter-binding classification picks up `Money total` parameter, resolves to composite via `ConventionDiscovery`, captures the inner-field list.

**Detection rules (per design Section 3):**
- Method parameter type resolves to MultiArgCtor via discovery.
- For each ctor parameter, derive the inner-binding name as `@{paramName}_{ctorArgName}` (e.g. `@total_Amount`, `@total_Currency`).
- The author's SQL is expected to reference those names (`WHERE Total_Amount = @total_Amount AND Total_Currency = @total_Currency`). No name-coupling enforcement — the generator just emits the unpacking; the SQL author owns the column-name match.

**TDD:**

**Commit:** `feat(generator): detect composite parameter binding`

### Task B.2 — Emit composite parameter binding

**Target emit:**

```csharp
// User declares: Task<int> UpdateAsync(Money total, CancellationToken ct)
// User's SQL:    "UPDATE Orders SET Amount = @total_Amount, Currency = @total_Currency"

var __p_total_Amount = __cmd.CreateParameter();
__p_total_Amount.ParameterName = "@total_Amount";
__p_total_Amount.Value = @total.Amount;
__cmd.Parameters.Add(__p_total_Amount);

var __p_total_Currency = __cmd.CreateParameter();
__p_total_Currency.ParameterName = "@total_Currency";
__p_total_Currency.Value = @total.Currency;
__cmd.Parameters.Add(__p_total_Currency);
```

**Implementation guidance:**
- Extend `EmitParameterBindingWithIndent` (or its underlying single-param helper) to recognise composite parameters and emit one `__p_{paramName}_{ctorArgName}` block per inner field.
- The unpacking accessor is the ctor-arg-name PascalCased into the C# property (e.g. `@total.Amount`). For records, this is automatic. For struct/class composites without auto-properties, may need `[Param]` annotation — defer to v0.6 if needed.

**TDD:**

## Phase C — Nullable composite handling (Days 5-6)

### Task C.1 — Emit nullable composite materialization (all-or-nothing)

**Design per Section 3.5:** `Money? Total` means:
- If ALL composite columns are DBNull → return `null`.
- If ANY are DBNull but not all → throw `ZeroAllocOrmMaterializationException` with a message naming which columns mixed.
- Otherwise → materialize normally.

**Target emit (scalar `Task<Money?>`):**

```csharp
var __amount_ord = __reader.GetOrdinal("Amount");
var __currency_ord = __reader.GetOrdinal("Currency");
var __amount_isNull = __reader.IsDBNull(__amount_ord);
var __currency_isNull = __reader.IsDBNull(__currency_ord);
if (__amount_isNull && __currency_isNull) return null;
if (__amount_isNull || __currency_isNull)
    throw new global::ZeroAlloc.ORM.ZeroAllocOrmMaterializationException(
        "Nullable composite 'Money' has mixed-null columns: Amount.isNull=" + __amount_isNull +
        ", Currency.isNull=" + __currency_isNull + ". All-or-nothing required.");
return new global::TestApp.Money(
    __reader.GetDecimal(__amount_ord),
    __reader.GetString(__currency_ord));
```

**Implementation guidance:**
- Use column-name `GetOrdinal` lookups (the v0.3-CLN1 backlog item flags that GetOrdinal-twice is a perf issue; this is a fresh emit so we hoist correctly from the start).
- For positional materialization (FlatRow nested composite), use positional indices instead of GetOrdinal.

**TDD:**
1. Snapshot: `Task<Money?>` standalone nullable composite.
2. Snapshot: `Task<OrderRow?>` where the inner is `record OrderRow(int Id, Money? Total)` (nullable composite NESTED in a non-nullable row).
3. Compile-smoke.

**Commit:** `feat(generator): emit nullable composite materialization (all-or-nothing DBNull check)`

### Task C.2 — Emit ZAO050 (warning) on nullable composite position

Per design Section 4: ZAO050 fires when a method uses a nullable composite type. The warning's purpose is to flag that mixed-null is a RUNTIME concern (not compile-time enforceable on Sqlite, which doesn't surface column nullability annotations).

**TDD:**

- Negative: `Task<Money> M()` (non-nullable) does NOT fire.

**Commit:** `feat(generator): emit ZAO050 on nullable composite positions`

**Commit:** `feat(generator): detect [Materialize(Factory)] annotation in classifier`

### Task D.2 — Emit factory invocation

**Target emit:**

```csharp
// User: [Materialize(Factory = "FromStorage")] record Money(decimal Amount, string Currency)
//       static Money FromStorage(string amountText, string currency) => new Money(decimal.Parse(amountText), currency);
//
// Emitted materialization at scalar position:
return global::TestApp.Money.FromStorage(
    __reader.GetString(0),     // factory's first param (amountText) ← matched by name
    __reader.GetString(1));    // factory's second param (currency)
```

**Implementation guidance:**
- Match factory parameters to columns by NAME (case-insensitive), not position.
- ZAO043 fires if factory not found.
- ZAO044 fires if factory parameter has no matching column or vice versa.
- The factory must be `static`. Non-static methods fail with a clear ZAO043 message.

**TDD:**
1. Snapshot: `Task<Money?>` with `[Materialize(Factory = "FromStorage")]` → emits FromStorage call.
2. Snapshot: nested in FlatRow — `record OrderRow(int Id, Money Total)` where Money carries the factory annotation.
3. ZAO043 fires when factory name doesn't exist.
4. ZAO044 fires when factory params don't match column conventions.

**Commit:** `feat(generator): emit [Materialize(Factory)] factory invocation`

## Risk register

| Risk | Mitigation |
|---|---|
| Composite column-index math leaks across emit shapes | Snapshot tests for nested-composite-in-FlatRow / DomainEntity / List exercise the flattened-count path. Tests cover the case where outer ctor arity ≠ flattened column count. |
| `[Materialize(Factory)]` parameter-to-column matching is name-based and case-sensitivity is ambiguous | Pick `OrdinalIgnoreCase` for case-insensitivity (matches Phase E v0.4 precedent). Document in ZAO044 doc + cookbook. |
| Nullable composite all-or-nothing check requires both IsDBNull calls regardless of value, costing a perf hit | Acceptable in v0.5 (correctness > microperf). File as v0.5-CLN if benchmark surfaces it. |
| Sqlite `decimal`-as-text breaks composite emit without `[Materialize(Factory)]` | The `FromStorage` recipe in Phase D.3 covers this. Update cookbook to recommend the factory pattern explicitly for Sqlite. |
| Recursive composites surface in adopter code | ZAO022 deferral message. Backlog item for v0.6+ if demand. |
| API overload during long agent dispatches | Cap each agent dispatch at ~5-6 commits. Phase A is 4 commits; B is 3; C is 3; D is 3; E is 3; F is 2. All within budget. |
