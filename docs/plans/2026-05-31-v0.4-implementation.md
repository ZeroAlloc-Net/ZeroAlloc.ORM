# ZeroAlloc.ORM v0.4 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `0.4.0` — `[Command]` attribute (NonQuery / Scalar / Identity), `[StoredProcedure]` attribute, named-tuple output parameters, ZAO060/061/062 diagnostics.

**Architecture:** Two new attribute consumers share the existing `[Query]` infrastructure (parameter binding, SQL preparation, connection lifecycle). The generator's `TransformMethod` learns to recognise `[Command]` and `[StoredProcedure]` alongside `[Query]`, and `ClassifyEmitShape` gains three new shapes — `CommandNonQuery`, `CommandScalar`, `CommandIdentity` — plus a `SprocOutputParams` materialization model for the named-tuple output-parameter case.

Provider-specific identity SQL (`SCOPE_IDENTITY()` vs `LASTVAL()` vs `RETURNING id`) is **author-supplied** in the SQL string — `Kind = CommandKind.Identity` only switches the emit from `ExecuteNonQueryAsync` to `ExecuteScalarAsync` + scalar materialisation. Automatic provider routing of the identity suffix is deferred to v2 (it would require runtime provider detection or compile-time pinning, both out of v0.4 scope).

**Tech Stack:** Same as v0.3. No new substrate work; `IAsyncDbCommand.ExecuteNonQueryAsync` and `ExecuteScalarAsync` already exist on the AdoNet.Async surface.

**Authoritative reference:** [`docs/design/2026-05-30-v1.0-design.md`](../design/2026-05-30-v1.0-design.md) Sections 2 + 2.5.

**Working backlog:** [`docs/plans/za-orm-backlog.md`](za-orm-backlog.md) — v0.4 entry (v0.4-T1 through T4).

**Pace:** ~2 weeks single-developer pace. ~28 tasks across 8 phases. TDD per task with snapshot + compile-smoke + integration tests where applicable.

**Lessons baked in (carried from v0.1 + v0.2 + v0.3):**

1. Snapshot test + compile-smoke test for every new emit shape. (Caught 3 bugs across the previous milestones.)
2. +15-20% per-task budget for analyzer friction (MA0048, MA0004, MA0006, CA1707, EPS06).
3. `@`-prefix verbatim safety on every emitted identifier reference.
4. No `Location`/`Compilation`/`SyntaxNode`/`ISymbol` in cached generator model types — `EquatableArray<T>` for collections.
5. Bundle related tasks per dispatch but cap at ~5-6 commits per agent run.
6. The integration suite drives correctness — every emit shape that lands must have at least one Sqlite round-trip test.

```csharp
public partial async global::System.Threading.Tasks.Task<global::TestApp.OrderRow?> GetOrderAsync(
    int @id, global::System.Threading.CancellationToken @ct)
{
    var __conn = @connection;
    var __openedHere = __conn.State != global::System.Data.ConnectionState.Open;
    if (__openedHere) await __conn.OpenAsync(@ct).ConfigureAwait(false);
```

## Phase E — Named-tuple output parameters (Days 9-10)

### Task E.1 — Detect named-tuple return on `[StoredProcedure]`

**Files:**
- Modify: `OrmGenerator.cs` — `ClassifyEmitShape` for `[StoredProcedure]` detects `Task<(T result, int newOrderId, ...)>` where:
  - Outer is `Task<>` / `ValueTask<>`.
  - Inner is a tuple with at least one element whose **field name** matches a `@param` name in the procedure signature (i.e. matches a method parameter name).
- Add `EmitShape.SprocWithOutputParams` (or augment `StoredProcedure` shape with an output-params flag).

**Detection rules:**
- For each tuple element, check if there's a corresponding C# parameter with the same name (case-insensitive match per ParamSet).
- If yes → it's an output parameter (becomes `Direction.Output` on the bound parameter).
- If no → it's the result row / scalar position (single result column, single row).
- At least one output param must be present, else it's just a regular sproc.

**TDD:** detection sentinel test for `Task<(OrderRow Result, int NewOrderId)>` where `NewOrderId` matches a `@newOrderId` parameter.

**Commit:** `feat(generator): detect named-tuple output parameters on [StoredProcedure]`

### Task E.2 — Emit output-parameter binding + read-back

**Files:**
- Modify: `OrmGenerator.cs` — `EmitSprocWithOutputParams` method.

**Target emit:**

```csharp
public partial async global::System.Threading.Tasks.Task<(global::TestApp.OrderRow Result, int NewOrderId)> RunAsync(
    int @customerId, int @newOrderId, global::System.Threading.CancellationToken @ct)
{
    var __conn = @connection;
    var __openedHere = __conn.State != global::System.Data.ConnectionState.Open;
    if (__openedHere) await __conn.OpenAsync(@ct).ConfigureAwait(false);
    try
    {
        await using var __cmd = __conn.CreateCommand();
        __cmd.CommandText = "usp_InsertOrder";
        __cmd.CommandType = global::System.Data.CommandType.StoredProcedure;

        var __p_customerId = __cmd.CreateParameter();
        __p_customerId.ParameterName = "@customerId";
        __p_customerId.Value = @customerId;
        __cmd.Parameters.Add(__p_customerId);

        var __p_newOrderId = __cmd.CreateParameter();
        __p_newOrderId.ParameterName = "@newOrderId";
        __p_newOrderId.Direction = global::System.Data.ParameterDirection.Output;
        __cmd.Parameters.Add(__p_newOrderId);

        await using var __reader = await __cmd.ExecuteReaderAsync(@ct).ConfigureAwait(false);
        if (!await __reader.ReadAsync(@ct).ConfigureAwait(false))
            throw new global::ZeroAlloc.ORM.ZeroAllocOrmMaterializationException("usp_InsertOrder returned no rows.");
        var __result = new global::TestApp.OrderRow(__reader.GetInt32(0), __reader.GetInt32(1), __reader.GetDecimal(2));

        // Drain reader so output parameters are populated (provider quirk).
        while (await __reader.ReadAsync(@ct).ConfigureAwait(false)) { }
        await __reader.DisposeAsync().ConfigureAwait(false);

        var __outNewOrderId = (int)__p_newOrderId.Value!;
        return (__result, __outNewOrderId);
    }
    finally
    {
        if (__openedHere) await __conn.CloseAsync().ConfigureAwait(false);
    }
}
```

**Critical:** the reader must be drained (and disposed before reading output parameters) on most providers (SQL Server, Sqlite, Postgres-via-Npgsql) because output parameters are only populated after the reader closes. The emit MUST do this explicitly.

**TDD:** snapshot + integration test (Sqlite supports output parameters via `CommandType.StoredProcedure` — see Microsoft.Data.Sqlite docs; if not, defer integration to Postgres v0.6).

**Commit:** `feat(generator): emit Direction=Output bindings + readback for [StoredProcedure] tuple returns`

### Task F.1 — ZAO060: `out`/`ref` parameter on async method

Actually redundant — C# already forbids `out`/`ref` on async methods (CS1988). But the design lists it. Verify whether the user would ever hit this case: if they try to declare a `[StoredProcedure]` method with an `out` parameter, the C# compiler rejects the source before the generator runs.

**Decision:** Skip ZAO060 unless we find a case the C# compiler doesn't already cover. Document in the diagnostic doc list as "intentionally not emitted; covered by CS1988." (Or emit it anyway as a friendlier message that points to named-tuple outputs.)

**TDD:** write the test FIRST, see if it can construct a passing-compiler source that triggers our diagnostic. If not → leave the descriptor reserved but unused. If yes → emit.

Tentative: reserve ZAO060 in `DiagnosticDescriptors.cs` + `AnalyzerReleases.Unshipped.md` + `docs/diagnostics/ZAO060.md` with a "reserved" note. Allows future expansion without renumbering.

## Risk register

| Risk | Mitigation |
|---|---|
| Sqlite doesn't support real stored procedures | Snapshot + compile-smoke prove the emit; integration round-trip deferred to Postgres v0.6 fixture if needed. Document the gap explicitly. |
| Output parameter readback requires reader-drain on some providers | Emit the drain unconditionally; one extra `while` loop is cheap insurance. |
| `Kind = Identity` ambiguity: SQL-supplied vs generator-supplied suffix | v0.4 docs commit to SQL-supplied. Provider routing is a v2 design decision. |
| Named-tuple field name matching: case-insensitivity vs ordinal | Match by case-insensitive name (`StringComparer.OrdinalIgnoreCase`) since C# allows `NewOrderId` ↔ `@newOrderId` mismatch and SQL parameter names are case-insensitive on most providers. Document the convention. |
| API overload during long agent dispatches (per v0.2 + v0.3 crashes) | Cap each agent dispatch at ~5-6 commits; Phases A+B and Phases C+D bundleable but split Phase E and Phase F. |
