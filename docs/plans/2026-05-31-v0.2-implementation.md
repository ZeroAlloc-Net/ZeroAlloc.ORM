# ZeroAlloc.ORM v0.2 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `0.2.0` — value-object discovery (ZA.ValueObjects integration via TypeConversions), enum round-trip (default int + `[StoreAsString]` re-introduction), multi-arg domain entity materialization with column-name-to-ctor-param resolution, 5 new diagnostics (ZAO040-044).

**Architecture:** Build out `ZeroAlloc.TypeConversions` from its current single-class state (PrimitiveCatalog only) to a real convention-discovery catalog consumed by the generator. The generator's `ClassifyEmitShape` + parameter-binding logic both call into the catalog. ZA.ValueObjects + record-single-arg + static-factory cases all funnel through the same discovery API.

**Tech Stack:** Same as v0.1. New dependency: `ZeroAlloc.ValueObjects` referenced by tests (to verify the `[ValueObject]` discovery branch); NOT a runtime dependency of ZA.ORM packages.

**Authoritative reference:** [`docs/design/2026-05-30-v1.0-design.md`](../design/2026-05-30-v1.0-design.md) Section 3 (discovery order). Anything contradicting the design needs re-check.

**Working backlog:** [`docs/plans/za-orm-backlog.md`](za-orm-backlog.md) — v0.2 entry. Items get crossed out when shipped.

**Pace:** Single developer, ~2 weeks. ~25 tasks. TDD throughout with the same patterns established in v0.1: snapshot tests, compile-smoke tests, integration tests.

**Lessons from v0.1 baked in:**

1. Every new emit shape ships with both a Verify-based snapshot test AND a `RunGeneratorAndCompile`-based compile-smoke test. Caught 3 real bugs in v0.1 that snapshots alone missed.
2. Per-task budget: implementer should plan for 15-20% analyzer friction (MA0048 file-per-type, MA0004 ConfigureAwait on await using, MA0006 string.Equals, CA1707 snake_case). The `tests/.editorconfig` already suppresses CA1707 for tests; the others need per-PR awareness.
3. `@`-prefix verbatim safety on every emitted identifier reference — already locked in for parameter names + CT names. Carries forward.
4. Plan-doc-as-you-go discipline — implementers commit plan-doc corrections alongside feature commits.

## Phase B — TypeConversions: build out `ConventionDiscovery` (Days 2-3)

### Task B.1: Define the `ConventionResult` API

**Files:**
- Create: `src/ZeroAlloc.TypeConversions/ConventionDiscovery.cs`
- Create: `src/ZeroAlloc.TypeConversions/ConventionKind.cs`
- Create: `src/ZeroAlloc.TypeConversions/ConventionResult.cs`

**Step 3: Implement** the public API per design doc Section 3:

```csharp
using Microsoft.CodeAnalysis;
using System.Collections.Immutable;

namespace ZeroAlloc.TypeConversions;

public static class ConventionDiscovery
{
    /// <summary>
    /// Resolves how a C# type should be materialized from a reader and bound to a SQL
    /// parameter. Returns a <see cref="ConventionResult"/> describing the strategy +
    /// the symbols the caller needs to emit the actual code.
    /// </summary>
    public static ConventionResult Resolve(ITypeSymbol type, ConventionContext context);
}

public sealed record ConventionResult(
    ConventionKind Kind,
    ISymbol? Factory,
    IPropertySymbol? Value,
    ImmutableArray<IParameterSymbol> ExpandedColumns);

public enum ConventionKind
{
    Unknown,
    Primitive,
    Enum,
    EnumAsString,
    ValueObject,
    StaticFactory,
    SingleArgCtor,
    MultiArgCtor,
    ExplicitMaterialize,  // for v0.5
}

public sealed record ConventionContext(Compilation Compilation);
```

(Adjust if `ISymbol`/`IPropertySymbol`/`IParameterSymbol` can't appear on the public API because TypeConversions is build-time-only and Roslyn-aware — this is fine.)

**Step 5: Commit** (`feat(typeconversions): scaffold ConventionDiscovery + ConventionResult API`).

### Task B.2: Implement built-in primitive discovery

Move the existing `PrimitiveCatalog` lookup into `ConventionDiscovery.Resolve` for the primitive case. Return `Kind = Primitive` for any type the existing `PrimitiveCatalog` recognises.

Tests: `tests/ZeroAlloc.TypeConversions.Tests/` — new project, mirrors the Abstractions.Tests shape. Test that `Resolve` returns `Primitive` for `int`, `string`, `decimal`, etc.

**Commit:** `feat(typeconversions): primitive discovery via PrimitiveCatalog passthrough`

### Task B.3: ZA.ValueObjects detection

**Approach:** Look for the `[ValueObject]` attribute on the type. ZA.ValueObjects' attribute is in the `ZeroAlloc.ValueObjects` namespace.

If found:
- Look for a `Value` property (the canonical wrapped-value accessor).
- Look for `static T From(TPrim)` factory.
- Return `Kind = ValueObject` with `Factory` = the static factory method symbol, `Value` = the property symbol.

Test sources need to reference `ZeroAlloc.ValueObjects` — add as a TEST-ONLY package reference to `ZeroAlloc.TypeConversions.Tests`. The runtime package does NOT depend on ZA.ValueObjects.

**Commit:** `feat(typeconversions): ZA.ValueObjects [ValueObject] discovery`

### Task B.4: Single-arg-ctor record discovery

Detect `record struct OrderId(int Value)` or `record class OrderId(int Value)` shapes. Match: type is a record, has exactly one public ctor, ctor takes exactly one parameter of a primitive type.

Returns `Kind = SingleArgCtor` with `Factory` = ctor symbol, `Value` = primary-ctor-synthesized `Value` property.

**Commit:** `feat(typeconversions): single-arg record discovery`

### Task B.5: Static factory discovery

Detect a `static T From(TPrim)` or `static T FromValue(TPrim)` method on the type. Same emit shape as ValueObject case.

**Commit:** `feat(typeconversions): static factory From/FromValue discovery`

### Task B.6: Enum discovery + `[StoreAsString]` detection

Detect `Kind = Enum` for any enum type. Detect `Kind = EnumAsString` when `[StoreAsString]` attribute is present on the enum type.

**Commit:** `feat(typeconversions): enum discovery + [StoreAsString] detection`

### Task B.7: Discovery order + ambiguity detection

Wire the discovery order per design doc Section 3:

```
1. [Materialize] explicit → ExplicitMaterialize (placeholder for v0.5)
2. Primitive (Task B.2)
3. ZA.ValueObjects (Task B.3)
4. Static factory (Task B.5)
5. Single-arg ctor record (Task B.4)
6. Enum / EnumAsString (Task B.6)
7. Multi-arg ctor (placeholder for v0.5)
8. Unknown → ZAO040
```

Each rule is checked in priority order. First match wins.

If two equal-priority rules match (rare — typically only when a user explicitly attributes a type with multiple conflicting conventions), return `Kind = Unknown` and let the diagnostic catch it via ZAO044.

**Commit:** `feat(typeconversions): wire convention discovery order + ambiguity guard`

### Task C.2: Emit value-object materialization

When a column's type resolves to `ValueObject`, emit:

```csharp
ColumnType.From(__reader.GetInt32(N))
```

Where `From` is the discovered factory and `GetInt32` is dictated by the underlying primitive.

**Snapshot test:** `ValueObjectMaterializationTests` — `record struct OrderId([property: ValueObject] int Value)` (or use ZA.ValueObjects' `[ValueObject]` attribute on the type itself per its docs).

Add to the test project: `<PackageReference Include="ZeroAlloc.ValueObjects" Version="1.*" />`.

**Snapshot:**

```csharp
public sealed record OrderRow(OrderId Id, decimal Total);

[Query("SELECT Id, Total FROM Orders LIMIT 1")]
public partial Task<OrderRow?> GetFirstAsync(CancellationToken ct);
```

Emit:
```csharp
return new global::TestApp.OrderRow(
    global::TestApp.OrderId.From(__reader.GetInt32(0)),
    __reader.GetDecimal(1));
```

**Compile-smoke test:** verify the emit compiles cleanly.

**Commit:** `feat(generator): emit value-object materialization via ZA.ValueObjects From factory`

### Task C.3: Emit value-object binding

When a parameter's type resolves to `ValueObject`, emit:

```csharp
__p_id.Value = @id.Value;
```

Reading the discovered `Value` property to unwrap.

**Snapshot test:** `ValueObjectParameterTests`.

**Commit:** `feat(generator): emit value-object parameter binding via Value property`

### Task C.4: Single-arg-ctor record support

Same emit shape as ValueObject case but uses the ctor instead of `From`:

```csharp

```

And binding:


**Snapshot test:** `SingleArgRecordTests`.

**Commit:** `feat(generator): emit single-arg-ctor record materialization + binding`

### Task C.5: Static factory support

**Commit:** `feat(generator): emit static-factory materialization`

## Phase E — Generator: multi-arg domain entity materialization (Days 9-10)

### Task E.1: Detect "class with named-param ctor" emit shape

Different from FlatRow positional records: a `class T` with a single public ctor whose params match column names. Column-name-to-ctor-param resolution via `reader.GetOrdinal("ColumnName")`.

**Detection:**
- Type is a `class` (not a record).
- Has exactly one public constructor with > 0 parameters.
- All ctor parameters resolve to known conventions (primitive, value-object, enum).

Add `EmitShape.DomainEntity` to the enum.

**Test source:**

```csharp
public sealed class Order
{
    public int Id { get; }
    public int CustomerId { get; }
    public decimal Total { get; }
    public Order(int id, int customerId, decimal total) => (Id, CustomerId, Total) = (id, customerId, total);
}

[Query("SELECT Id, CustomerId, Total FROM Orders WHERE Id = @id")]
public partial Task<Order?> GetByIdAsync(int id, CancellationToken ct);
```

**Emit:**

```csharp
return new global::TestApp.Order(
    __reader.GetInt32(__reader.GetOrdinal("Id")),
    __reader.GetInt32(__reader.GetOrdinal("CustomerId")),
    __reader.GetDecimal(__reader.GetOrdinal("Total")));
```

Note: `GetOrdinal` is used for column-name lookup (vs positional). This is the key distinction from FlatRow.

**Snapshot + compile-smoke + integration tests.**

**Commit:** `feat(generator): emit DomainEntity materialization with column-name-to-ctor-param resolution`

## Phase F — Diagnostics ZAO040-ZAO044 (Days 11-12)

### Task F.1: ZAO040 — no resolvable construction strategy

When `ConventionDiscovery.Resolve` returns `Unknown` for a materialization-target type, emit `ZAO040` Error.

**Test:** materialization to a type that doesn't match any convention (e.g. `class Order` with no ctor + no factory).

**Commit:** `feat(generator): emit ZAO040 when no construction strategy resolves`

### Task F.2: ZAO041 — no resolvable unwrap strategy (binding side)

When a method parameter's type doesn't resolve to a bindable convention (no `Value` property, no primitive, no enum), emit `ZAO041` Error.

**Commit:** `feat(generator): emit ZAO041 when no binding strategy resolves`

### Task F.3: ZAO042 — `[StoreAsString]` on non-enum

Type-level diagnostic. If `[StoreAsString]` appears on a non-enum, emit `ZAO042` Error.

**Implementation:** ZA.ORM.Generator scans every type that the consumer marks `[StoreAsString]` (not just enums) and fires if any are non-enum. Roslyn pipeline: `SyntaxProvider.ForAttributeWithMetadataName("ZeroAlloc.ORM.StoreAsStringAttribute", ...)`.

**Commit:** `feat(generator): emit ZAO042 when [StoreAsString] appears on non-enum type`

### Task F.4: ZAO043 — `[Materialize(Factory = "X")]` missing method (placeholder for v0.5)

Add the descriptor. Don't wire the detection yet — that's v0.5 work when `[Materialize]` is re-introduced.

**Commit:** `feat(generator): reserve ZAO043 descriptor for v0.5 [Materialize] support`

### Task F.5: ZAO044 — ambiguous discovery

When `ConventionDiscovery.Resolve` returns `Unknown` AND multiple rules matched equally (per Task B.7), emit `ZAO044` Error with helpful text about which rules matched.

**Test:** a type that both has a `Value` property AND a `static From` AND no clear precedence.

In practice this rarely fires because the discovery order is strict; this diagnostic is the safety net.

**Commit:** `feat(generator): emit ZAO044 when convention discovery is ambiguous`

## What v0.2 deliberately does NOT do

- Multi-column composite materialization (`Money(decimal Amount, string Currency)`) — v0.5.
- `[Materialize(Factory)]` custom resolution — v0.5.
- `[Command]` + `[StoredProcedure]` attributes — v0.4 (re-add).
- `[Materialize]` attribute — v0.5 (re-add).
- Multi-result-set + `IAsyncEnumerable<T>` streaming — v0.3.

Re-add of stripped attributes happens in the milestone that implements them. `StoreAsString` is the only one re-added in v0.2.
