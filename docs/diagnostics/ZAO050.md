# ZAO050 тАФ Nullable composite type requires runtime all-or-nothing check

**Severity:** Warning
**Category:** ZeroAlloc.ORM

## Status

**Active** тАФ fires for every method that uses a nullable composite type at a return position, a parameter position, or as a nested ctor parameter on a FlatRow / DomainEntity.

## What is a nullable composite

A composite is a type with a multi-arg public constructor whose parameters each resolve to a primitive / value-object / enum / single-arg-ctor / static-factory convention (no recursive composites). The canonical example:

```csharp
public readonly record struct Money(decimal Amount, string Currency);
```

A *nullable* composite is the same type used with the `?` annotation:

* `Task<Money?>` (scalar return)
* `record OrderRow(int Id, Money? Total)` (nested in a FlatRow)
* `Task<int> UpdateAsync(Money? total, CancellationToken ct)` (parameter)

## Why does it warn

The composite's underlying SQL columns (`Amount`, `Currency`) are individually nullable on the database side. The C# nullable annotation can't statically prove the schema declares them NOT NULL **together**. The generator therefore enforces the contract at runtime:

* **All composite columns DBNull** тЖТ return `null`.
* **Any (but not all) DBNull** тЖТ throw `ZeroAllocOrmMaterializationException` with a message naming every column and its DBNull state.
* **Otherwise** тЖТ materialize normally.

The mirror image on the bind side (parameter `Money? total`):

* **`total is null`** тЖТ every inner `DbParameter` value is `DBNull.Value`.
* **`total` non-null** тЖТ standard positional unpacking via `total.Value.Amount` / `total.Value.Currency`.

ZAO050 flags every position where this runtime contract applies so adopters consciously opt into the semantics.

## Suppression

If your schema guarantees the composite columns are always populated or always null together (a single NOT-NULL / NULL constraint applied to all of them as a group), the runtime check is a no-op for your data тАФ suppress via one of:

**Project-level (recommended for adopters who have audited the whole schema):**

```xml
<PropertyGroup>
  <NoWarn>$(NoWarn);ZAO050</NoWarn>
</PropertyGroup>
```

**Source-level pragma** (per-method scoping тАФ works when `TreatWarningsAsErrors` is OFF, or when the pragma is paired with `<WarningsNotAsErrors>ZAO050</WarningsNotAsErrors>` at the project level):

```csharp
#pragma warning disable ZAO050
[Query("SELECT Amount, Currency FROM Orders WHERE Id = @id")]
public partial Task<Money?> GetTotalAsync(int id, CancellationToken ct);
#pragma warning restore ZAO050
```

Source-generator-emitted diagnostics carry an `ExternalFile` location that `#pragma warning disable` directives do not scope-suppress when `TreatWarningsAsErrors` is enabled at the project level. The project-level `<NoWarn>` is the more reliable adopter-facing mechanism.

## Diagnostic message

```
Method '<MethodName>' uses nullable composite type '<TypeFullName>' at <position>. The all-or-nothing DBNull check is enforced at runtime; partial-null columns throw ZeroAllocOrmMaterializationException at materialize time (or send DBNull-for-all on bind time). If your schema guarantees these columns are populated or null together, suppress this warning with #pragma warning disable ZAO050.
```

`<position>` is one of:

* `return position` тАФ scalar `Task<Money?>`.
* `nested in return type` тАФ nullable composite inside a FlatRow / DomainEntity ctor.
* `parameter '<paramName>'` тАФ `Money? total` parameter.
