# ZAO008 — Multi-statement SQL with single-result return type

**Severity:** Error
**Category:** ZeroAlloc.ORM

## Trigger

Fires when a `[Query]`-annotated method's SQL text contains a `;` separator (multiple statements) but the return type is a single-result shape like `Task<int>` or `Task<Order>`. Multi-statement queries must surface results as a tuple so each statement's result set has a destination.

The statement counter is string-literal aware: `;` characters inside single-quoted (`'...'`) or double-quoted (`"..."`) SQL literals are ignored, including the SQL escape forms `''` and `""`. A trailing `;` followed only by whitespace also does not introduce a new statement.

> Note: v0.1 does not yet emit multi-result tuple shapes (deferred to a later milestone). This diagnostic ships as a pre-emptive guard so user code does not silently lose statements.
>
> TODO(v0.2): the counter does not yet understand SQL line comments (`--`), block comments (`/* */`), or PostgreSQL dollar-quoted strings (`$tag$…$tag$`). A `;` inside any of those still counts as a statement separator today; a proper SQL tokenizer lands in v0.2.

## `[Command]` exemption

ZAO008 fires for `[Query]`-annotated methods only. `[Command]`-annotated methods are explicitly exempt because legitimate command shapes routinely span multiple statements:

* **Identity (`Kind = Identity`)** — providers without a `RETURNING` clause surface the inserted key via a second statement, e.g. SQLite's `SELECT last_insert_rowid()` or SQL Server's `SELECT SCOPE_IDENTITY()`.
* **Multi-step DML** — a single `[Command]` may legitimately bundle related writes (an INSERT followed by an UPDATE against a join table, for example) when both share one logical transaction boundary.

The `[Command]` shapes (`NonQuery`, `Scalar`, `Identity`) each have their own well-defined single-result contract that does not depend on statement count — `ExecuteScalarAsync` simply reads the first column of the last result set.

### Example — `[Command]` Identity that does NOT fire ZAO008

```csharp
public sealed partial class Repo(IAsyncDbConnection connection)
{
    // Two statements in one [Command] is fine — the second produces the
    // identity value that ExecuteScalarAsync reads.
    [Command(
        "INSERT INTO Orders (CustomerId) VALUES (@customerId); SELECT last_insert_rowid();",
        Kind = CommandKind.Identity)]
    public partial Task<int> InsertOrderAsync(int customerId, CancellationToken ct);
}
```

## Fix

Either remove the extra statement(s) so the SQL is a single query, or change the return type to a tuple matching one result per statement.

## Example

```csharp
// Fires ZAO008:
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [Query("SELECT 1; SELECT 2;")]
    public partial Task<int> GetOneAsync(CancellationToken ct);  // only first survives
}

// Fix (single-statement):
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [Query("SELECT 1")]
    public partial Task<int> GetOneAsync(CancellationToken ct);
}
```
