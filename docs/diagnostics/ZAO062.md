# ZAO062 — Named-tuple field does not match any parameter

**Severity:** Warning
**Category:** ZeroAlloc.ORM

## Trigger

Fires when a `[StoredProcedure]`-annotated method returns a named tuple in
which:

1. **At least one** tuple field name matches a C# parameter (case-insensitive)
   — proving the **named-tuple output-parameter pattern** is in use, AND
2. **At least one OTHER** tuple field name does NOT match any C# parameter.

The non-matching field is treated as a **result-set column** by the generator
(the existing multi-result-set classification rules apply). This is a
legitimate shape — for example, a sproc that returns both a result set and an
output parameter — but it's also the shape produced by a common author
mistake: typing the tuple field name wrong silently demotes an intended
**output parameter** to a result column at runtime.

Surfacing the field name in the diagnostic message lets the adopter check
their intent before shipping.

## Fix

Two options, depending on intent:

* **Intended as an OUTPUT parameter.** Rename the tuple field to match the
  parameter name (case-insensitive). For example, parameter `newOrderId`
  should pair with tuple field `NewOrderId`.
* **Intended as a RESULT-SET column.** The current behaviour is correct — the
  field is bound to a column from the procedure's result set. Suppress the
  warning with `#pragma warning disable ZAO062` if the shape is intentional
  and the field name is too useful to change.

## Example

```csharp
// Fires ZAO062 — `Total` matches no parameter and is treated as a result
// column. If the author intended `Total` as an output parameter, they meant
// to add `int total` to the signature.
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_InsertOrder")]
    public partial Task<(int NewOrderId, int Total)> InsertAsync(
        int customerId, int newOrderId, CancellationToken ct);
}

// Fix 1 — output intent. Add `total` to the parameter list:
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_InsertOrder")]
    public partial Task<(int NewOrderId, int Total)> InsertAsync(
        int customerId, int newOrderId, int total, CancellationToken ct);
}

// Fix 2 — result-column intent. Acknowledge the shape:
#pragma warning disable ZAO062
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_InsertOrderWithTotalsRow")]
    public partial Task<(int NewOrderId, int Total)> InsertAsync(
        int customerId, int newOrderId, CancellationToken ct);
}
#pragma warning restore ZAO062
```

## Not triggered when

* **Zero matches.** The output-params pattern isn't in use; the tuple flows
  through the regular `MultiResultSet` path and ZAO062 is not applicable.
* **All matches.** Every tuple field matches a parameter — the output-only
  sub-case, no result-set columns; nothing to warn about.
* **Non-`[StoredProcedure]` methods.** `[Query]` and `[Command]` tuple returns
  stay on the `MultiResultSet` path. A tuple field name happening to match a
  C# parameter is coincidence there (parameter binding is by name, not by
  tuple-field rebinding), and ZAO062 would be noise.
