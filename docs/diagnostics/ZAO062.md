# ZAO062 — Named-tuple field does not match any parameter

**Severity:** Warning
**Category:** ZeroAlloc.ORM

## Trigger

Fires when a `[StoredProcedure]`-annotated method returns a named tuple in
which:

1. **At least one** tuple field name matches a C# parameter (case-insensitive)
   — proving the **named-tuple output-parameter pattern** is in use, AND
2. **At least one OTHER** tuple field name does NOT match any C# parameter.

The non-matching field is treated as a **result-set column** by the generator
(the existing multi-result-set classification rules apply). This is a
legitimate shape — for example, a sproc that returns both a result set and an
output parameter — but it's also the shape produced by a common author
mistake: typing the tuple field name wrong silently demotes an intended
**output parameter** to a result column at runtime.

Surfacing the field name in the diagnostic message lets the adopter check
their intent before shipping.

## Fix

Two options, depending on intent:

* **Intended as an OUTPUT parameter.** Rename the tuple field to match the
  parameter name (case-insensitive). For example, parameter `newOrderId`
  should pair with tuple field `NewOrderId`.
* **Intended as a RESULT-SET column.** The current behaviour is correct — the
  field is bound to a column from the procedure's result set. Suppress the
  warning with `#pragma warning disable ZAO062` if the shape is intentional
  and the field name is too useful to change.

## Example

```csharp
// Fires ZAO062 on `Total` — there are TWO non-matching tuple fields
// (`Result` and `Total`) plus a matching `NewOrderId`. Heuristic 1 treats
// the first non-matching field (`Result`) as the conventional result-row
// position and skips it; ZAO062 fires on `Total` (the second non-matching
// field), which is more likely a typo than a legitimate result position.
// If the author intended `Total` as an output parameter, they meant to add
// `int total` to the signature.
public sealed record OrderRow(int Id, int CustomerId, decimal Total);

public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_InsertOrder")]
    public partial Task<(OrderRow Result, int Total, int NewOrderId)> InsertAsync(
        int customerId, int newOrderId, CancellationToken ct);
}

// Fix 1 — output intent. Add `total` to the parameter list so `Total`
// matches a parameter and becomes an output position:
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_InsertOrder")]
    public partial Task<(OrderRow Result, int Total, int NewOrderId)> InsertAsync(
        int customerId, int newOrderId, int total, CancellationToken ct);
}

// Fix 2 — result-column intent. Acknowledge the shape:
#pragma warning disable ZAO062
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_InsertOrderWithTotalsRow")]
    public partial Task<(OrderRow Result, int Total, int NewOrderId)> InsertAsync(
        int customerId, int newOrderId, CancellationToken ct);
}
#pragma warning restore ZAO062
```

The canonical `(OrderRow Result, int NewOrderId)` shape with one matching
parameter and one non-matching field does NOT fire ZAO062 — `Result` is
treated as the conventional result-row position and skipped (Heuristic 1).

## Not triggered when

* **Zero matches.** The output-params pattern isn't in use; the tuple flows
  through the regular `MultiResultSet` path and ZAO062 is not applicable.
* **All matches.** Every tuple field matches a parameter — the output-only
  sub-case, no result-set columns; nothing to warn about.
* **Exactly one non-matching field.** The first non-matching tuple field is
  treated as the conventional result-row position (e.g. `(OrderRow Result,
  int NewOrderId)` where `Result` is the row, not a typo'd output) and
  skipped. ZAO062 fires only on the second-and-later non-matching field —
  see **Limitations** below.
* **Non-`[StoredProcedure]` methods.** `[Query]` and `[Command]` tuple returns
  stay on the `MultiResultSet` path. A tuple field name happening to match a
  C# parameter is coincidence there (parameter binding is by name, not by
  tuple-field rebinding), and ZAO062 would be noise.

## Limitations

ZAO062 trades two known false-negatives for low noise on the canonical
sproc-with-outputs shape:

1. **Heuristic 1 (first non-matching field skipped).** The canonical
   `(OrderRow Result, int NewOrderId)` pattern has one non-matching field
   (`Result`) which is conventionally the result-row position. ZAO062 would
   fire on every adopter's first sproc-with-outputs if it didn't skip this
   position, so the diagnostic only triggers on the **second**-and-later
   non-matching tuple field. A single mistyped output (`(int NewOrderId,
   int Tota1)` where `Tota1` was meant to be `Total` matching a `total`
   parameter) WILL slip through — there is exactly one non-matching field,
   and it gets skipped. Multi-typo cases still surface.
2. **Typo in the parameter name itself.** ZAO062 only fires when at least
   one tuple field matches a parameter (the `SprocWithOutputParams` shape
   was selected). If the typo lives on the **parameter** side (tuple
   `NewOrderId` + parameter `newOredrId`), zero fields match, routing
   falls through to the `MultiResultSet` shape, and ZAO062 is never
   evaluated. The adopter sees a runtime mismatch instead of a build-time
   warning.

Both limitations are deliberate v0.4 trade-offs. A future analyzer pass may
fold parameter-side typo detection into a separate diagnostic.
