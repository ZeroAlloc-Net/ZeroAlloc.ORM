# ZAO051 — Factory parameter does not match any SELECT column

**Severity:** Error
**Category:** ZeroAlloc.ORM

## Status

**Active** — fires in v0.5 Phase D for `[Materialize(Factory = "X")]` factory dispatches when name-based matching is enabled and one of the factory parameters does not match any candidate column name (case-insensitive).

## Trigger

When a type carries `[Materialize(Factory = "X")]` and:

- The generator can statically resolve a candidate set of column names for the position the factory dispatch is being built at (for example, a composite type nested inside a `DomainEntity` row uses the underlying composite ctor parameter names PascalCased as the candidate set), AND
- One of the factory's parameter names does NOT match any candidate name with `OrdinalIgnoreCase`.

The matching contract is **exact case-insensitive equality** — adopters who want alias names use SQL `SELECT Amount AS amountText`.

### Example trigger

```csharp
[Materialize(Factory = "FromStorage")]
public readonly record struct Money(decimal Amount, string Currency)
{
    // 'rawAmount' does not match any of {Amount, Currency} — ZAO051 fires.
    public static Money FromStorage(string rawAmount, string currency)
        => new Money(decimal.Parse(rawAmount), currency);
}

public sealed record Cart(int Id, Money Total);

public sealed partial class Repo(IAsyncDbConnection connection)
{
    [Query("SELECT Id, Amount, Currency FROM Carts WHERE Id = @id")]
    public partial Task<Cart> GetByIdAsync(int id);
}
```

## Fix

Pick one of:

1. **Rename the factory parameter** so its name (case-insensitively) matches a candidate column name. Above: `string rawAmount` → `string amount`.
2. **Use SQL `AS` aliasing** to surface a column name that matches the factory parameter. Above: `SELECT Id, Amount AS rawAmount, Currency FROM Carts WHERE Id = @id`.
3. **Use positional matching** — drop the `[Materialize(Factory)]` annotation and rely on the type's underlying ctor or a positional `static From(...)` factory via convention discovery.

## Fallback (positional matching)

When the generator CANNOT statically derive a candidate column set (the FlatRow positional path / composite-at-scalar path where no SQL column names are available at classification time), ZAO051 is suppressed and **positional** factory-param-to-column matching is used as the documented fallback. The factory's parameter list drives the read positions in declaration order.

## Suppression

ZAO051 surfaces a real adopter-facing modeling defect — silently misbinding columns into the wrong factory parameter is materially worse than a compile-time error. Suppressing this diagnostic via `<NoWarn>` / pragma is not recommended; prefer one of the three fixes above.
