# ZAO064 — [StoredProcedure(Batch=...)] is ignored

**Severity:** Info
**Category:** ZeroAlloc.ORM

## Trigger

Fires when a method is annotated with `[StoredProcedure(..., Batch = X)]`
where `X` is anything other than `BatchMode.Never` (the sproc pipeline's
default).

The `Batch` named argument exists on `StoredProcedureAttribute` only for
**symmetry** with `QueryAttribute` and `CommandAttribute`. Stored
procedures encapsulate their own batching semantics server-side: the
generator always emits a single `DbCommand` with
`CommandType.StoredProcedure`, and multi-result-set sprocs are walked
via `NextResultAsync` in the materializer regardless of `Batch`. There
is no `;`-joined script for `IAsyncDbBatch` to usefully decompose.

The non-default value is therefore **silently ignored at emit time**.
ZAO064 surfaces the no-op at compile time so adopters do not ship a
misleading attribute and discover the discrepancy later.

## Fix

Two options:

* **Remove the `Batch` named argument.** The default
  (`BatchMode.Never`) already matches the emit behaviour.
* **Set it explicitly to `BatchMode.Never`.** Same behaviour, but
  preserves the explicit intent in the source. ZAO064 will not fire.

## Example

```csharp
public sealed partial class Repo(IAsyncDbConnection connection)
{
    // Fires ZAO064 — Batch is ignored on sprocs.
    [StoredProcedure("usp_GetOrder", Batch = BatchMode.Always)]
    public partial Task<OrderRow?> GetOrderAsync(int id, CancellationToken ct);
}

// Fix 1 — remove the Batch override:
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_GetOrder")]
    public partial Task<OrderRow?> GetOrderAsync(int id, CancellationToken ct);
}

// Fix 2 — explicit Never (same emit, preserves intent):
public sealed partial class Repo(IAsyncDbConnection connection)
{
    [StoredProcedure("usp_GetOrder", Batch = BatchMode.Never)]
    public partial Task<OrderRow?> GetOrderAsync(int id, CancellationToken ct);
}
```

## Not triggered when

* The method uses `[Query]` or `[Command]` — `Batch` is honoured on
  those pipelines (see `ResolveBatchStrategy`).
* The `Batch` named argument is omitted on `[StoredProcedure]` — the
  attribute's default (`BatchMode.Never`) is exactly what the emit
  produces, so there is nothing to warn about.
* The adopter explicitly writes `Batch = BatchMode.Never` — same
  semantics as the default; no behavioural drift to surface.
