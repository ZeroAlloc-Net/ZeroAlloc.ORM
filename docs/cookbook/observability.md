# Observability — composing ZA.ORM with ZA.Telemetry

ZeroAlloc.ORM ships **no `ActivitySource`** of its own. By design, observability
lives at the **consumer seam** via
[ZeroAlloc.Telemetry](https://github.com/ZeroAlloc-Net/ZeroAlloc.Telemetry)'s
`[Instrument]` pattern. This recipe shows how to compose the two source
generators so OpenTelemetry spans and metrics wrap every database call without
either library knowing about the other.

## Why no built-in `ActivitySource`?

Three reasons drove the v0.6 design pivot away from an always-on, ZA.ORM-emitted
`ActivitySource`:

1. **AOT footprint.** A built-in source pulls every adopter onto the
   `System.Diagnostics.DiagnosticSource` cost even when they don't subscribe.
   Keeping ZA.ORM slim means consumers pay only for what they use.
2. **Cross-cutting concern, application boundary.** Tracing, metrics, and
   logging are application concerns — the application chooses the exporter,
   the sampling policy, and the span naming convention. ZA.ORM emitting spans
   would impose all three.
3. **Clean composition.** ZA.Telemetry already solves
   `[Trace]` / `[Count]` / `[Histogram]` on arbitrary interfaces. ZA.ORM emits a
   partial method body that fills the interface. The two generators target
   completely different code shapes — one wraps from the outside (proxy
   class), the other fills from the inside (partial body) — so they compose
   without any coupling.

ZA.ORM-only adopters who don't want telemetry simply don't declare an
interface. The cost is zero.

## The composition pattern

The shape is **interface + partial implementation + DI wiring**. ZA.Telemetry
generates a sealed proxy class that wraps the implementation; ZA.ORM fills the
SQL-execution body of the implementation.

```csharp
using System.Data.Async;
using System.Threading;
using System.Threading.Tasks;
using ZeroAlloc.ORM;
using ZeroAlloc.Telemetry;

public sealed record OrderRow(int Id, int CustomerId, decimal Total);

// 1. Declare the interface — this is what ZA.Telemetry hooks.
[Instrument("MyApp.Orders")]
public interface IOrderRepository
{
    [Trace("orders.get_by_id")]
    [Count("orders.fetched")]
    [Histogram("orders.get_by_id_ms")]
    Task<OrderRow?> GetByIdAsync(int id, CancellationToken ct);
}

// 2. Implement it as a ZA.ORM partial class.
public sealed partial class OrderRepository(IAsyncDbConnection connection)
    : IOrderRepository
{
    [Query("SELECT Id, CustomerId, Total FROM Orders WHERE Id = @id")]
    public partial Task<OrderRow?> GetByIdAsync(int id, CancellationToken ct);
}
```

At compile time:

- `ZeroAlloc.ORM.Generator` fills the `OrderRepository.GetByIdAsync` partial
  with the SQL-execution body (open connection, bind `@id`, read row,
  materialise to `OrderRow?`).
- `ZeroAlloc.Telemetry.Generator` emits a separate sealed class —
  `OrderRepositoryInstrumented : IOrderRepository` — in the same namespace.
  Each method that carries `[Trace]` / `[Count]` / `[Histogram]` is wrapped in
  the corresponding span / counter / histogram recording, with the call
  delegated to an inner `IOrderRepository`.

The two generators never touch the same syntactic slot: ZA.Telemetry writes
into a new file (the `Instrumented` proxy), ZA.ORM writes into the partial of
your implementation type. **There is no collision risk** between them.

## What the generated proxy looks like

For the interface above, ZA.Telemetry emits something equivalent to:

```csharp
// <auto-generated />
internal sealed class OrderRepositoryInstrumented : IOrderRepository
{
    private static readonly ActivitySource _activitySource = new("MyApp.Orders");
    private static readonly Meter _meter = new("MyApp.Orders");
    private static readonly Counter<long> _orders_fetched =
        _meter.CreateCounter<long>("orders.fetched");
    private static readonly Histogram<double> _orders_get_by_id_ms =
        _meter.CreateHistogram<double>("orders.get_by_id_ms");

    private readonly IOrderRepository _inner;
    public OrderRepositoryInstrumented(IOrderRepository inner) => _inner = inner;

    public async Task<OrderRow?> GetByIdAsync(int id, CancellationToken ct)
    {
        using var activity = _activitySource.StartActivity("orders.get_by_id");
        var start = Stopwatch.GetTimestamp();
        try
        {
            var result = await _inner.GetByIdAsync(id, ct);
            _orders_fetched.Add(1);
            _orders_get_by_id_ms.Record(Stopwatch.GetElapsedTime(start).TotalMilliseconds);
            return result;
        }
        catch (Exception ex)
        {
            activity?.SetStatus(ActivityStatusCode.Error, ex.Message);
            _orders_get_by_id_ms.Record(Stopwatch.GetElapsedTime(start).TotalMilliseconds);
            throw;
        }
    }
}
```

Exact emit shapes are subject to ZA.Telemetry's version — see
[ZA.Telemetry attribute reference](https://github.com/ZeroAlloc-Net/ZeroAlloc.Telemetry/blob/main/docs/attributes.md)
for the canonical output. The contract is: a sealed wrapper class living in
the same namespace as the interface, named `{TypeName}Instrumented` with any
leading `I` stripped (so `IOrderRepository` becomes `OrderRepositoryInstrumented`).

## Wiring at the application boundary

Both generators have done their job at compile time. At runtime you wire DI so
the application reaches `IOrderRepository` through the instrumented proxy:

```csharp
// Program.cs
var builder = WebApplication.CreateBuilder(args);

// OpenTelemetry — match the source name in [Instrument(...)].
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing => tracing
        .AddSource("MyApp.Orders")
        .AddOtlpExporter())
    .WithMetrics(metrics => metrics
        .AddMeter("MyApp.Orders")
        .AddOtlpExporter());

// Db connection — the IAsyncDbConnection abstraction the partial uses.
builder.Services.AddScoped<IAsyncDbConnection>(_ =>
    new SqliteConnection(connString).AsAsync());

// Register the concrete ZA.ORM implementation, then expose IOrderRepository
// through the ZA.Telemetry-generated wrapper.
builder.Services.AddScoped<OrderRepository>();
builder.Services.AddScoped<IOrderRepository>(sp =>
    new OrderRepositoryInstrumented(sp.GetRequiredService<OrderRepository>()));

var app = builder.Build();
app.MapGet("/orders/{id}", async (int id, IOrderRepository repo, CancellationToken ct) =>
    await repo.GetByIdAsync(id, ct));
app.Run();
```

Application code depends on `IOrderRepository`, never on the concrete
partial. The `Instrumented` proxy is transparent — adopters can swap it out
(register `OrderRepository` directly as `IOrderRepository`) to opt out of
telemetry in, say, integration test environments.

## Attribute reference (quick recap)

| Attribute                  | Effect                                                            | Recorded on                          |
| -------------------------- | ----------------------------------------------------------------- | ------------------------------------ |
| `[Instrument("source")]`   | Triggers the generator. Names the `ActivitySource` + `Meter`.     | The interface itself (required).     |
| `[Trace("span.name")]`     | Wraps the call in `ActivitySource.StartActivity(...)` + try/catch. | Every invocation; Error on throw.    |
| `[Count("metric.name")]`   | Increments a `Counter<long>` by 1.                                | Success path only.                   |
| `[Histogram("metric.ms")]` | Records elapsed milliseconds in a `Histogram<double>`.            | Every invocation (success and throw). |

Full reference:
[ZA.Telemetry attribute docs](https://github.com/ZeroAlloc-Net/ZeroAlloc.Telemetry/blob/main/docs/attributes.md).

## Adding tags / parameters to spans

ZA.Telemetry v1 generates spans with no per-call tags by default — the span
name is the only thing varying per invocation. If you need to surface
parameter values (e.g. `id = 42`) as span attributes, the canonical pattern is
to enrich inside the partial method body — but ZA.ORM owns the partial, so
that path is closed. Two options:

1. **Wrap manually for cases that need rich tags.** Drop `[Trace]` on that one
   method, write a hand-rolled `ActivitySource.StartActivity(...)` block in a
   regular (non-partial) method on the repository, and call the
   ZA.ORM-generated partial from inside it.
2. **Add tags at the call site.** Use `Activity.Current?.SetTag("order.id", id)`
   in the application code that calls `IOrderRepository`. The current
   `Activity` is the one ZA.Telemetry started — adding tags from outside
   works because spans are mutable until disposed.

For pure span/counter/histogram telemetry without per-call tags, the
out-of-the-box composition is enough.

## Metrics-only or tracing-only

The four attributes are independent. A method with only `[Trace]` records
spans but no metrics. A method with only `[Count]` + `[Histogram]` records
metrics but no spans. A method with **no** ZA.Telemetry attributes is passed
through to the inner implementation without any wrapping — useful for
high-frequency helper methods on the interface where instrumentation overhead
isn't worth it.

## When NOT to use this composition

- **Driver-level instrumentation already covers your needs.** Packages like
  `Npgsql.OpenTelemetry` emit spans at the ADO.NET wire level — every
  command, every parameter binding. If that granularity is what you want,
  prefer the driver instrumentation. ZA.Telemetry composition wraps at the
  *repository* level: one span per logical operation, regardless of how many
  underlying commands it issues. The two can coexist; wire-level spans become
  children of the repository span.
- **You're inside a hot loop where 200 ns of overhead matters.** ZA.Telemetry
  benchmarks at ~200 ns per call with no listeners attached (parity with
  hand-written instrumentation). For most workloads that's noise; for
  per-row hot loops in ETL paths, consider skipping `[Trace]` on the inner
  call and wrapping the outer batch operation instead.
- **You don't have an OpenTelemetry pipeline set up.** Without a listener,
  `ActivitySource.StartActivity` returns `null` and the cost collapses to a
  field load + null check — but you also get no telemetry data. Either wire up
  exporters or skip the `[Instrument]` interface.

## AOT compatibility

Both generators are AOT-safe by design:

- ZA.ORM emits raw ADO.NET — no reflection, no `params object[]`, closed
  generic types at generation time.
- ZA.Telemetry emits a closed proxy class — no reflection, no
  `MakeGenericType`, static `ActivitySource` / `Meter` field initializers.

Composing the two under `<PublishAot>true</PublishAot>` is expected to
produce no IL2026 / IL3050 warnings. An automated AOT collision smoke test
(`tests/ZeroAlloc.ORM.TelemetryCollision.AotSmoke`) that would gate this
composition end-to-end on every PR is **deferred** — the v0.6 attempt
surfaced a ZA.Telemetry-side nullable-annotation bug in its
`InstrumentGenerator` (CS8613 / CS8603 on `Task<T?>` return types through
the generated proxy). The smoke will be re-attempted once ZA.Telemetry
ships a generator update that preserves nullable annotations across the
`[Instrument]` boundary (tracked as v0.6-CLN1 in the backlog). The
conceptual composition pattern documented above is unaffected — only the
automated AOT verification is pending.

## Diagnostics

The ZA.ORM × ZA.Telemetry composition produces no specific diagnostics
today. Each generator emits its own diagnostic catalog independently:

- ZA.ORM: `ZAO0xx` codes — see [`../diagnostics/`](../diagnostics/).
- ZA.Telemetry: see the
  [attribute reference](https://github.com/ZeroAlloc-Net/ZeroAlloc.Telemetry/blob/main/docs/attributes.md)
  for placement rules.

If both generators emit cleanly on their own files, they compose. If one
fails, the other still runs — diagnostics from the failing generator point
exclusively at its own attributes.
